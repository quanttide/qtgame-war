# prototype.html 分析：从设计文档到可运行原型的 AI 展开过程

## 概述

`docs/index.md` 是一份极其精简的概念文档（29行），定义了战棋游戏的核心框架：动机、三种玩法层次、四类建模元素。`prototype.html`（379行）是一个完全可交互的 HTML 原型。从概念到原型，AI 在多个维度进行了大量"展开"。

---

## 一、叙事层的展开

### docs 只提供了动机
> "还原真实战场指挥官的视角……明知道历史走向，却依然感到步步惊心、决策艰难"

### prototype 构建了完整叙事
- **标题**："战棋游戏 - 指挥官视角"，直接响应 docs 动机
- **副标题**："还原真实战场指挥官的视角，体验'明知道历史走向，却依然步步惊心'的压迫感"——几乎逐字翻译 docs 动机为产品标语
- **具体战场设定**：东北方向敌军装甲营、山地/森林地形、回合12（暗示已进行到中局）
- **单位命名**："敌装甲""敌步兵""我步兵""我装甲""我炮兵"——创造了具体的对立双方

**AI 的展开方式**：将抽象动机转化为具体叙事场景。动机中的"历史走向"变成了一局进行到一半的游戏（turn 12），"决策艰难"变成了三条各有代价的路线。

---

## 二、玩法层的展开

### docs 定义了两种模式
1. 简单：互动剧本模式（判断→命令→结算）
2. 复杂：兵棋推演模式（自定义排兵部署）

### prototype 实现了三层结构
| 标签页 | 对应 docs 玩法 | 具体实现 |
|--------|---------------|---------|
| **情报** | 判断环节 | 三种类型情报（可信/存疑/已证伪）+ 后勤报告 |
| **命令** | 命令+结算 | 3条预设路线 + 自定义(AI参谋) + 执行结果弹窗 |
| **兵棋** | 兵棋推演模式 | 可用单位列表 + 地形说明 + 开始推演按钮 |

### AI 的关键展开
1. **将"判断"具体化为情报系统**：docs 只说"判断"，AI 设计了三级情报可信度（可信→存疑→证伪），对应真实战场的情报不确定性——这是从玩法机制反向推动叙事可信度。
2. **将"命令"具体化为选项系统**：docs 只说"用选项 A、B、C 和自定义等下达"，AI 为每条命令配了**完整战术描述 + 风险提示**（如"伤亡预计15-20%"），并在执行后给出**叙事化结果 + 数据化反馈**（控制率变化）。
3. **将"结算"具体化为结果弹窗**：docs 只说"是否掌握战场主动权"，AI 实现了三个结果：success/fail 两种状态、叙事描述、数据变化（控制率更新、回合递增）、地图状态暗示（占领区域、敌军动向）。

## 三、交互层的展开

### docs 没有定义任何交互细节

### prototype 设计了完整的交互流程
1. **Tab 切换**（情报/命令/兵棋）
2. **命令选择 → 显示执行栏 → 下达命令 → 弹窗结果 → 更新面板 → 关闭重置**
3. **UI 状态反馈**：选中高亮、hover 变色、显示/隐藏切换、弹窗遮罩

### AI 的交互设计思路
- **渐进式披露**：执行栏默认隐藏，选命令后才显示——避免信息过载
- **统一的状态管理**：`selectedCommand`、`currentTurn` 等全局变量驱动 UI 状态
- **视觉一致性**：所有选中态用 `border-color: #f78166`，提示色统一（绿色=可信/成功、黄色=存疑/风险、红色=证伪/失败）

---

## 四、视觉层的展开

### docs 没有定义任何视觉风格

### prototype 采用了完整的设计系统
- **色彩系统**：暗色主题（`#0d1117` 背景），橙色（`#f78166`）为主色——高对比、军事风格
- **布局系统**：左面板（320px 情报/命令/兵棋）+ 右区域（六角格地图）+ 顶部标题栏
- **组件风格**：卡片式、圆角 3px、1px 边框、左色彩条指示
- **字体层级**：24px 标题 / 14px 标签 / 13px 正文 / 11-12px 辅助信息

### AI 的视觉决策
- 选择暗色主题而非亮色——暗色更符合"战场指挥所"的氛围
- 选择橙色为主色而非军绿色——橙色在暗色背景上辨识度更高，且橙色在 UI 中常表示"警告/注意"（符合战场的不确定性氛围）
- 信息密度控制：每个情报框/命令框的信息量限制在 2-4 行，保持可扫读性

## 五、技术层的展开

### docs 只提了一句建模
> 格子：六边形单元格 | 棋盘：由符合真实战场比例的单元格构成

### prototype 实现了完整六角格引擎
- **数学计算**：六边形顶点坐标（6个顶点 60° 间隔）、轴向坐标转像素坐标、行偏移排列（奇偶行错开 50%）
- **尺寸参数**：`HEX_SIZE=32`，列宽 `COL_W = HEX_W * 0.75`，行高 `ROW_H * 0.75`
- **地形系统**：plain(平原)/mountain(山地)/forest(森林)/water(水域)，随机分布（12%/16%/5%/67%）
- **视野系统**：`revealed` 布尔值控制迷雾/揭示状态（约 60% 已揭示）
- **单位系统**：单位对象 `{name, type}` 附着在具体格子上

### AI 的技术决策
- **纯前端单文件**：选择无依赖的原生 HTML/CSS/JS，无需构建工具即可运行——降低验证成本
- **SVG 渲染六边形**：用 SVG polygon 而非 Canvas，方便 CSS 控制样式和事件
- **伪随机地图**：`Math.random()` 每次刷新不同，但单位位置固定——保证每次都有新挑战但核心场景一致

## 六、文档未提及但 AI 主动添加的内容

| 新增内容 | 归属维度 | 设计意图 |
|---------|---------|---------|
| **回合计数器** | 叙事+玩法 | 让玩家感知时间流逝，每次执行命令回合 +1 |
| **战场控制率** | 玩法反馈 | 数据化玩家的决策效果，68%→72% 等变化提供成就感 |
| **情报模糊度** | 玩法机制 | 直观反映"战争迷雾"强度，从 UI 层强化不确定性 |
| **自定义命令(AI参谋)** | 玩法扩展 | 增加"玩家输入"维度，不只是选 ABC——虽然尚未实现输入接口 |
| **后勤报告** | 叙事深度 | 补充"补给线""弹药""机动受限"等细节，增强真实感 |
| **结果中的成败提示色** | 反馈设计 | 绿色=成功、红色=失败，一目了然 |

## 七、总结：AI 的展开方法论

从 `docs/index.md` 到 `prototype.html`，AI 的展开遵循了以下模式：

1. **每句展开为子系统**：docs 的一行"判断→命令→结算"变成了三个独立子系统（情报/命令/结果）
2. **抽象概念具体化**："信息干扰"→三种类型情报（可信/存疑/证伪），"战场主动权"→68%→72% 数字变化
3. **补充 gap**：docs 未提及的视觉风格、交互细节、技术实现、数据结构被 AI 自主补充
4. **叙事与机制相互强化**：每个玩法机制都配了叙事解释（"敌军侦察连遭重创，情报获取能力下降"），防止机制显得抽象空洞
5. **预留扩展点**：兵棋模式只做入口和描述（功能开发中）、自定义命令只做了选项框架、地图可以支持拖拽部署——为后续开发预留空间

核心结论：AI 的"展开"不是机械地逐条实现文档需求，而是**理解文档背后的设计意图**，然后以该意图为约束条件，在叙事、视觉、交互、技术四个层面同时进行创造性发散与收敛，最终形成一个在每个维度都自洽、且彼此之间互相增强的完整原型。

## 八、原型评估：五项标准逐项评分

### 8.1 评估框架

| 维度 | 权重 | 考察内容 |
|------|------|---------|
| 意图传达度 | 20% | 用户不读 docs 能否理解"这是什么游戏" |
| 交互闭环率 | 30% | 每个操作是否有完整的 触发→反馈→状态更新 链 |
| 信息层级清晰度 | 15% | 首屏 3 秒能否回答"我在哪/我能做什么/当前状态" |
| 视觉-机制一致性 | 20% | 颜色/样式是否服务于含义，而非纯装饰 |
| 可扩展性预留 | 15% | 原型是否是"死胡同"，是否为后续开发留了空间 |

### 8.2 逐项评估

#### 意图传达度 — A-(90/100)

| 考察项 | 结果 | 说明 |
|--------|------|------|
| 标题传递核心体验 | ✅ | "指挥官视角"直接点明角色，"战棋游戏"交代类型 |
| 副标题解释独特性 | ✅ | "明知道历史走向，却依然步步惊心"精准传达了 core loop |
| 首屏 3 秒理解 | ⚠️ | 布局清晰，但"情报模糊度:高"等术语需要领域知识 |

**扣分点**：没有 onboarding/引导，新用户第一次打开不知道应该先看情报还是先点命令。

#### 交互闭环率 — B+(82/100)

| 命令 | 闭环链路 | 状态 |
|------|---------|------|
| A/B/C | 点击→选中高亮→执行栏出现→点"下达命令"→弹窗结果→关闭→面板数据更新→回合+1 | ✅ 完整 |
| custom | 同上，但无输入接口，只能用预设文案 | ⚠️ 伪闭环 |
| 兵棋模式 | 点"开始推演"→alert 弹窗→无后续 | ❌ 断链 |

**扣分点**：
- 自定义命令承诺"AI 参谋生成方案"，实际打开后无输入框，直接选就能执行——制造了预期落差
- 地图不能交互（`cursor: default`，无点击/悬停事件），与左侧命令系统的决策没有视觉联动（选了 A 后地图没有高亮变化）
- 执行命令后地图不更新——结果说了"占领(3,3)(3,4)"，但六角格上没有任何变化

#### 信息层级清晰度 — A(95/100)

| 层级 | 内容 | 评价 |
|------|------|------|
| L0 全局 | 顶部标题 + 左面板 + 右地图 | 一目了然 |
| L1 左侧 | 三个 Tab（情报/命令/兵棋），当前活跃态橙底高亮 | 清晰 |
| L2 内容 | 情报按可信度排序（可信→存疑→证伪），命令 A/B/C 按激进程度排序 | 合理 |
| L3 详情 | 每条命令含标签+描述+风险，视觉分离清晰 | 优秀 |

唯一问题：情报面板的"后勤报告"放在最后，它的可信度标记是[可信]，但视觉上是普通情报框样式（无绿条），与其可信状态不匹配。

#### 视觉-机制一致性 — B+(85/100)

| 颜色 | 含义 | 一致？ |
|------|------|--------|
| `#f78166` 橙 | 主要行动、选中态、玩家单位 | ✅ 一致贯穿 |
| `#3fb950` 绿 | 可信/成功 | ✅ 贯穿 |
| `#f85149` 红 | 已证伪/失败/敌方 | ❌ 不一致：敌方单位文字是红色，但敌方单位所在的六角格描边是橙色(`has-unit` 用了 `#f78166`) |
| `#ffd700` 黄 | 存疑/风险 | ✅ 贯穿 |

**扣分点**：`hex-cell.has-unit` 的 `stroke: #f78166` 无法区分敌我——地图上敌我单位都是橙色边框，只能靠文字颜色分辨，而文字在地图缩小时几乎不可读。

#### 可扩展性预留 — B+(83/100)

| 预留项 | 状态 | 评价 |
|--------|------|------|
| 六角格数据结构 | `{terrain, revealed, unit}` — 结构化可增删字段 | ✅ |
| 兵棋模式入口 | 留有 UI 框架+地形数值（防御+30% 等） | ✅ 但功能未实现 |
| 自定义命令 | 选项框架已留，缺输入组件 | ⚠️ 半成品 |
| 地形系统 | 4 种地形 + 颜色映射，扩展只需加颜色表 | ✅ |
| AI 参谋 | 叙事上承诺了"生成方案"，代码里只是硬编码 | ⚠️ 叙事承诺 > 技术实现 |
| 地图交互 | 无事件绑定（click/hover/drag），需从头加 | ❌ |
| 多回合循环 | 回合 counter 实现了，但地图不随回合变化 | ⚠️ |

| 意图传达度 | 20% | 90 | 18.0 |

### 8.4 核心结论

**强项**：意图传达和视觉系统是最大亮点——打开页面 3 秒就能感受到"这是一个战争指挥官决策游戏"。信息层级干净，没有不必要的装饰。

**最大短板**：交互闭环有三处断裂（自定义命令无输入、地图无响应、兵棋模式是空壳）。作为"原型"可以接受，但如果目标是验证玩法假设，地图不响应命令结果会让测试者无法判断"这个机制是否有潜力"。

**最高优先级改进**：地图状态与命令结果的联动——哪怕只是简单的高亮被占领格子，都能让交互闭环率达到可测试的水平。
