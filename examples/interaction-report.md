# 交互设计评估报告 — prototype.html

## 一、核心交互模型

```
┌─────────────────────────────────────────────────────┐
│                   用户意图                           │
│  "我想了解战场局势" → 情报Tab → 读4条情报           │
│  "我想做决策"       → 命令Tab → 选路线 → 下达       │
│  "我想看结果"       → 弹窗叙事 → 更新数据面板       │
│  "我想验证效果"     → 地图……地图没变化 → 困惑        │
└─────────────────────────────────────────────────────┘
```

当前交互模型是一个**线性瀑布流**：情报→决策→执行→反馈。问题是**反馈回路断裂**——用户收到结果后无法在地图上验证。

---

## 二、交互闭环审计

### 2.1 完整闭环 ✅

| 步骤 | 触发 | 反馈 | 状态更新 |
|------|------|------|---------|
| Tab切换 | 点击Tab | Tab高亮切换，内容面板切换 | active class 转移 |
| 选择命令 A/B/C | 点击命令卡片 | 卡片选中态(橙色边框+背景变深)，执行栏滑出 | `selectedCommand` 赋值 |
| 下达命令 | 点击"下达命令"按钮 | 弹窗出现，展示结果叙事 | 控制率更新，回合+1 |
| 关闭结果 | 点击"继续" | 弹窗消失，执行栏隐藏，选中态清除 | `selectedCommand` 置空 |

### 2.2 断裂/半成品 ❌⚠️

| 步骤 | 问题 | 严重度 | 影响 |
|------|------|--------|------|
| **地图响应** | 执行命令后地图无任何变化 | 🔴 阻断级 | 用户无法验证"结果"，核心反馈回路断裂 |
| **自定义命令** | 承诺"输入战术意图→AI生成方案"，实际无输入框，点即执行 | 🟡 信任级 | 制造预期落差，降低对系统的信任 |
| **兵棋模式** | 点"开始推演"→alert→无后续 | 🟡 功能级 | 暴露了未完成的功能，破坏完整感 |
| **情报→命令过渡** | 情报Tab和命令Tab之间无上下文延续 | 🟢 流畅级 | 用户读了情报后切到命令，之前看到的情报信息不直观关联到决策 |

## 三、注意力模型分析

### 3.1 当前视线路径（"选命令→看结果"完整操作）

```
① 地图(审视战场) ──→ ② 切到命令Tab ──→ ③ 左面板选路线
                                                        ↓
⑥ 地图(想验证) ←── ⑤ 弹窗(读结果) ←── ④ 执行栏点下达
```

**5 次区域跨越** > 健康的 3 次上限。每次跨越都要求用户重新定位视线焦点。

### 3.2 注意力竞争

```
┌─ 左面板(320px) ──────────────────┬─ 右地图区域(剩余宽度) ─────┐
│                                   │                            │
│  [情报] [命令] [兵棋]  ←Tab行    │  回合:12  主动权:我方      │
│  ┌─────────────────┐              │  控制:68% 模糊度:高        │
│  │ 侦察报告 [可信]  │             │                            │
│  │ 敌军主力在东北…  │             │    六角格地图              │
│  ├─────────────────┤             │    (9x11 网格)             │
│  │ 模糊情报 [存疑]  │             │                            │
│  │ 南侧可能有…      │             │    [敌] [敌]               │
│  ├─────────────────┤             │                            │
│  │ 干扰信息 [证伪]  │             │        [我][我][我]        │
│  │ (5,5)区域为空…  │             │                            │
│  ├─────────────────┤             │                            │
│  │ 后勤报告 [可信]  │             │                            │
│  │ 补给线畅通…      │             │                            │
│  └─────────────────┘             │                            │
└───────────────────────────────────┴────────────────────────────┘
```

左面板和右地图在视觉权重上**完全平等**——用户不知道该把注意力放在哪里。理想设计中，两个区域应有一个主次关系（如地图占 70% 视觉权重）。

## 四、交互问题的根本原因

### 4.1 布局决定行为

当前布局是**编辑器风格**（左工具面板 + 右工作区），但游戏需要的是**叙事风格**（沉浸式中央区域 + 上下文敏感的操作入口）。这两个模式相互冲突：

| 维度 | 编辑器风格（当前） | 叙事风格（游戏需要） |
|------|-------------------|-------------------|
| 信息呈现 | 并排展示，用户自己筛选 | 逐步揭示，引导用户注意力 |
| 操作路径 | 面板操作→右侧看效果 | 在地图上操作→在上下文里做决策 |
| 反馈方式 | 文字+数字更新 | 地图状态变化+叙事描述 |

### 4.2 状态可视化空白

系统内部维护了多个游戏状态：`hexData`（格子数据）、`currentTurn`、`selectedCommand`。但地图的渲染是**一次性的**——`createHexGrid()` 只在页面加载时调用一次。执行命令后虽然更新了数据面板，但从未调用地图的重新渲染。这是最简单的修复点。

### 4.3 自定义命令的预期管理失败

自定义命令的设计模式是"先承诺能力，再展示 UI"，正确模式应该是"先展示 UI，再使用能力"。用户看到"AI参谋"时预期的是一个输入界面，但实际得到的只是一个可点击的卡片。

## 五、改进建议（按优先级）

### P0 — 修复核心反馈回路

**问题**：执行命令后地图不更新。
**改动量**：在 `executeCommand()` 末尾调用地图重渲染函数（更新格子颜色/单位位置/revealed 状态）。
**效果**：完成"决策→结果→验证"闭环，用户能在地图上看到"我占领了(3,3)"。

### P1 — 消除注意力竞争

**方案 A（推荐）**：地图占满全宽，左面板改为**浮动/折叠式**，默认隐藏，点击"情报/命令"时滑出。
**方案 B（轻量）**：左面板宽度从 320px 缩到 260px，去掉情报框的独立卡片边框，改用列表样式。

### P2 — 降低 GCW 信息块数

**方案**：4 张回合卡片折叠为 2 行，或整合到地图顶部的 overlay 条中。情报 4 框合并为 1 个容器内的 4 条文本（去掉 4 套独立边框+标题）。

### P3 — 自定义命令补充输入接口

在 `cmd-option` 点击后展开一个 `<textarea>` + "提交"按钮，AI 参谋的回复直接填入结果弹窗。只需在 `selectCmd('custom')` 分支中追加 HTML 渲染。

## 六、总结

| 维度 | 评价 |
|------|------|
| 交互模型 | 线性瀑布流，基本合理但反馈回路断裂 |
| 闭环完整性 | 主路径完整，地图响应缺失导致核心验证失败 |
| 注意力管理 | 左右等权分裂，GCW 超标 70%，用户不知道该看哪 |
| 预期管理 | 自定义命令承诺 > 实现，信任受损 |
| 改进性价比 | P0 修复（地图联动）只需 3-5 行代码，换来最大的体验提升 |

一句话结论：原型验证了"决策→叙事反馈"的玩法是成立的，但**验证链断在地图上**——修复地图状态同步后，这个原型才真正"可测试"。
