# 领域层

领域层包含规则和数据模型，不依赖任何 UI 框架。

## 概念划分

| 概念 | 内容 | 为什么独立 |
|------|------|-----------|
| 兵种 | 兵种模板（攻击/防御/移动力）和战场实例（位置/血量/状态） | 模板不可变可共用，实例每局创建随战局变化 |
| 地形 | 坐标系统（六角格几何、寻路）和地形效果（移动消耗、防御加成） | 坐标是通用数学算法，地形效果是游戏数据 |
| 战斗 | 纯计算，输入攻守双方和战场环境，输出命中/伤害/文本 | 不改任何数据，调用方拿到结果后自己应用 |
| 战役 | 随时间变化的值：战力、工事强度、增援标记 | 这些值在战斗中不断被修改，供战斗和 UI 查询 |
| 引擎 | 把上述概念串起来：计算移动范围、攻击目标、生成援军、判定胜负 | 不保存状态，每次调用传参，纯逻辑编排 |

## 两类数据：配置与状态

开战前定死的叫配置（地图坐标、兵种属性、援军回合），存 JSON。战斗中变化的叫状态（战力、血量、位置），存内存字段。

配置只在开局读一次。状态每回合被代码改。

## 为什么状态直接改

兵棋的操作模式是"用户操作 → 算结果 → 更新画面"，一帧内完成，不存在并发。直接改字段够用，`copyWith` 重建对象多此一举。将来如果要回放，应该在操作层记日志，而不是把模型全改成不可变。
