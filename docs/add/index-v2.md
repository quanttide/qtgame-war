# 架构设计文档

以下是一个脱离具体编程语言与前端框架的通用架构设计思路，专门针对“六边形战棋推演引擎”及其衍生的战役级系统。它提炼自我们之前的所有实现，旨在提供一套可复用的抽象，无论你用 React、Flutter、Unity 还是 C++ 实现，这套思路都适用。

—

1. 核心理念：三层解耦

整个系统被划分为三个严格分离的层次：

```
┌──────────────────────────────────────┐
│          表现层 (Presentation)       │
│  - 画布渲染                          │
│  - UI控件（面板、按钮、日志）        │
│  - 输入处理（鼠标/触摸/手势）        │
└──────────────┬───────────────────────┘
               │ 只通过事件/状态订阅通信
┌──────────────▼───────────────────────┐
│          调度层 (Orchestration)       │
│  - 状态容器                          │
│  - 事件分发                          │
│  - 副作用管理（AI 延迟、动画）       │
└──────────────┬───────────────────────┘
               │ 调用纯函数
┌──────────────▼───────────────────────┐
│           领域层 (Domain)             │
│  - 游戏规则（移动、攻击、视野）      │
│  - 地图与坐标系统                    │
│  - 战斗结算                          │
│  - AI 决策逻辑                       │
│  - 全局战役参数（体力、增援等）      │
└──────────────────────────────────────┘
```

核心原则：领域层完全不知道表现层的存在，调度层只知道“状态变化需要通知视图”，表现层只负责将状态映射为像素和交互。这保证了你可以在不修改任何游戏逻辑的情况下，更换渲染方式或从 2D 升级到 3D。

—

2. 领域层设计（纯逻辑，无 UI）

这是整个引擎的心脏，所有规则都封装在此。它应该是一个无状态的规则执行器，每次调用都输入当前状态和待执行的动作，输出新的状态。

2.1 核心数据模型

· Unit：单位实体，包含 ID、阵营、坐标、基础属性（攻击/防御/移动/射程）、当前血量、特殊标签、是否已行动。
· Map：地形网格（二维数组），每个格子存储地形类型枚举。地形定义移动消耗、防御加成、视野遮盖、特殊效果（如核心工事）。
· CampaignState：全局战役变量，如部队体力、编制、敌工事强度、增援倒计时、崩溃风险等。它们可能影响所有单位的攻防属性。
· GameState：聚合了 Unit 列表、Map、CampaignState、当前回合、阶段（玩家行动/AI/结束）、选中状态、高亮范围等。

所有这些模型都是不可变数据结构（immutable），状态更新时生成新对象，避免副作用。

2.2 坐标系统与寻路

· 采用标准的Odd-r 偏移坐标系或立方坐标，并提供如下纯函数接口：
  · pixel_to_hex(x, y) -> Hex
  · hex_to_center(Hex) -> Point
  · hex_distance(a, b) -> int
  · neighbors(Hex) -> List<Hex>
· 移动范围计算：基于移动力、地形消耗、单位阻挡、特殊地形准入限制的BFS 搜索。该函数接受一个单位和一个地图，返回可达格子集合。
· 攻击范围计算：基于攻击距离和战争迷雾（未揭示敌军不可选为目标）以及掩体穿透规则，返回可攻击的格子集合。

2.3 战斗结算器

纯函数：resolve_combat(attacker, defender, terrain, campaign_state) -> CombatResult

· 命中概率 = 基础概率 + 攻击方修正（含全局体力/编制惩罚） - 地形防御修正 - 距离衰减。
· 伤害计算考虑突击、爆破等特殊规则。
· 返回结果包含：是否命中、伤害值、被消灭信息、可能触发的战役效果（如工事强度下降）。
· 结算后不直接修改单位状态，而是返回一个描述变化的数据结构，由上层决定如何应用到状态中。

2.4 AI 行为策略

将 AI 决策同样视为纯函数：ai_act(state) -> List<Action>

· 根据当前可见敌军单位进行威胁评估（优先级：突击单位 > 高攻击单位 > 低血量单位）。
· 动作分为攻击和向特定目标移动，均使用与玩家相同的移动/攻击合法性校验。
· 对于援军，编写特殊的推进策略（朝某个关键坐标移动）。
· AI 输出的 Action 列表再由引擎逐个应用到状态，形成多步动画。

3. 调度层设计（状态管理）

调度层负责持有当前 GameState，提供修改状态的唯一入口，并通知视图更新。

3.1 单向数据流 + 事件驱动

· 视图层产生用户意图事件（如 SELECT_UNIT、MOVE_TO、ATTACK、END_TURN）。
· 调度层接收事件，调用领域层函数得到新的状态，并以不可变方式替换当前状态。
· 副作用（如 AI 延迟执行、动画）由调度层通过回调或异步机制管理，但领域层永远是同步的。
· 状态更新后，调度层将新状态推送给所有订阅的视图，或由视图主动拉取差异。

3.2 时间推进与回合控制

· 回合制结构：玩家行动阶段 → 玩家结束回合 → AI 行动阶段 → 回合数 +1，重新开启玩家阶段。
· 每个阶段内部，单位只能行动一次，行动后标记 hasActed。
· 全局时间效果（如体力自然衰减、增援计时）在回合切换时统一处理。

3.3 日志与事件溯源

· 每次状态变更都可以附带一份日志条目（LogMessage），包含类型（信息/命中/紧急）和文本。
· 日志本身也是状态的一部分，视图据此渲染。
· 可选择实现事件溯源（Event Sourcing）：所有操作记录为事件序列，便于回放或网络同步。

—

4. 表现层设计（视图与交互）

表现层负责两件事：渲染当前状态和将用户输入转化为事件。

4.1 渲染器抽象

· 定义渲染器接口 IRenderer，暴露 render(state) 方法。
· 具体实现可以是 Canvas 2D、SVG、WebGL、Flutter CustomPaint 等，只需遵循接口。
· 渲染逻辑分解为：
  1. 遍历所有六边形，根据地形类型和高亮标记绘制底色和描边。
  2. 绘制地形图标（文字或矢量符号）。
  3. 绘制单位（形状、兵种符号、血条、行动标记）。
  4. 绘制战争迷雾遮盖（未揭示敌军显示“?”）。
  5. 绘制战役结束覆盖层。
· 为提高性能，渲染器可以只重绘脏区域，或者在状态未变化时跳过绘制。

4.2 输入适配器

· 定义输入适配器接口 IInputAdapter，负责接收原始点击/触摸坐标，并将其转换为游戏事件。
· 流程：屏幕坐标 → pixel_to_hex() → 结合当前选中状态和合法高亮范围 → 决定动作类型 → 生成事件。
· 支持右键或长按取消选中。
· 多指手势（缩放/平移）可以独立处理，不干扰游戏逻辑，只需调整视图矩阵。

4.3 UI 组件化原则

· 将信息面板拆分为多个观察者组件，每个只依赖必要的状态切片：
  · 回合信息：currentTurn、phase
  · 单位卡片列表：playerUnits、selectedUnitId
  · 全局状态：campaignState（体力、工事、增援）
  · 作战日志：logs
· 组件本身不包含任何游戏逻辑，点击按钮时直接向调度层发送对应事件。
· 所有动态样式（如高亮、禁用、文字颜色）均基于状态计算，不存储本地状态。

5. 可扩展性设计

5.1 多尺度兼容

正如我们之前的讨论，同一个引擎可以通过配置数据来模拟从排级到军级的不同尺度：

· 地图描述文件：定义格子代表的实际距离（30米 → 3公里），地形定义相应调整。
· 单位模板：定义单位类型（步兵排、炮兵连、团、师）的基础属性。
· 规则参数表：命中率公式、移动消耗、视野距离等全部外部化，方便调整。

你可以创建多个剧本（Scenario），每个剧本包含：

· 地图文件（地形二维数组和坐标修正）
· 初始单位部署（位置、属性、阵营）
· 全局战役参数（时间限制、援军触发回合、特殊规则）
· 胜利条件函数（可替换）

引擎本身不限定剧本，只负责根据剧本初始化状态。

5.2 AI 行为树/策略模式

AI 不应硬编码为 if-else 链。抽象出 ICommander 接口，包含 generate_actions(state) -> List<Action> 方法。

· 防御性 AI：固守核心，被动反击。
· 进攻性 AI：主动向玩家薄弱点移动并攻击。
· 事件驱动 AI：检测到援军到达后切换策略。

5.3 网络对战扩展

由于状态是纯数据且事件驱动，引入网络层只需：

· 终端捕获本地事件，发送至服务器。
· 服务器持有权威游戏状态，应用事件并同步给双方。
· 延迟隐藏可通过简单的客户端预测 + 状态回滚实现（框架已预留事件序列）。

6. 技术无关的接口定义示例（伪代码）

```
// 领域层核心函数签名
function calculate_move_range(unit, map, all_units) -> Set<Hex>
function calculate_attack_targets(unit, map, all_units) -> Set<Hex>
function resolve_combat(attacker, defender, map, campaign) -> CombatResult
function apply_result(state, combat_result) -> GameState
function check_victory(state) -> VictoryCondition?

// 调度层接口
class GameSession {
    state: GameState
    event_handler(event): 
        new_state = state_engine.apply(state, event)
        state = new_state
        notify_renderers()
        if new_state.phase == AI: schedule_ai_steps()
}

// 表现层接口
interface IBoardRenderer {
    render(state)
}
interface IInputHandler {
    on_tap(local_position) -> Event
}
```

7. 总结：一张蓝图

无论你使用什么技术栈，最终的系统都可以归结为一张高内聚低耦合的模块蓝图：

· 数据（Map, Unit, CampaignState）：定义游戏世界的静态与动态事实。
· 规则（移动/攻击/AI/胜利条件）：用纯函数表达世界如何变化。
· 调度（状态管理、事件循环）：控制变化的发生时机和顺序。
· 表现（渲染、输入、音效）：将世界投射给玩家，并接受玩家指令。

各层通过不可变数据结构通信，这样，你完全可以在不修改规则的前提下，将整个项目从 Web 迁移到 Unity，或者在同一个项目中同时拥有 2D 和文本界面版本。这套架构就是我们之前从田花园排级突击，一路抽象到帝丘店战役历程中所沉淀下来的核心设计。
