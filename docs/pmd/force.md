# PMD: 部队模块

## 原型验证（examples/prototype/situation.html）

这个原型在验证四个核心设计假设：

**1. 三维权衡机制** — 进攻（消耗体力）vs 休整（恢复体力但敌军迫近）vs 撤退（放弃任务）。时间是不可逆的稀缺资源，三个按钮构成一个没有最优解的三角困境。

**2. 叙事状态呈现** — 体力、编制、敌军距离等数值不直接暴露，而是通过三团长报告和侦察回报的叙事文本呈现。验证"信息不对称"在交互层面是否成立。

**3. 选项推演** — 战前 UI 展示每个选项的预期后果（"进攻→体力约 X，极可能崩溃"），让玩家在信息不完全的情况下做有依据的决策。这正是 PRD 里"不是菜单选择器，是立场宣誓器"的交互落地。

**4. 失败即叙事** — 崩溃或任务失败时不弹 game over，而是以叙事覆盖层呈现（"各团失联，防线不复存在"），验证"失败本身成为故事一部分，而非中断"。

## 文档映射

与三份文档的对应关系：

**PRD（核心体验）**
- 它 > 不是 → 验证了"不是全知视角，是信息不对称下的决策模拟"。玩家看不到体力精确值，只能听到"还能站着的不到一个连"
- 动态选项过滤器 → 撤退按钮文案随任务状态改变（"放弃阵地" vs "任务已完成，可撤离"）

**QA（验证规格）**
- C02 · 我的部队真的累了 → 进攻按钮带风险警告，体力跌破阈值时提示"极可能崩溃"
- C03 · 战场在我不在的时候活着 → 时间自动推进，敌军持续迫近，玩家不动世界也在变
- C08 · 我第一次发现自己不能撤退 → 退路被断（retreatRoute = false），撤退变成溃退

**架构设计（分层映射）**
- 验证了领域规则引擎独立的可行性 — Commander 类就是 Domain Layer 的雏形：纯 JS 类，不涉及 DOM，包含红线校验、概率计算、后果推演
- 数据流验证了 CommanderBloc → 红线校验 → UI 更新的单向链路

简而言之：原型在 PRD 定义的核心体验指导下，集中验证 QA 中 C02/C03/C08 三个关键场景，并用纯领域类验证架构分层的前端可行性。
