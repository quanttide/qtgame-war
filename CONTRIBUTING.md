# 贡献指南

## 工作流程选择

在动手之前，先判断任务在哪个层级：

| 信号 | 层级 | 流程 |
|------|------|------|
| 只改 `src/` | 实现层 | 直接动手，改完提交 |
| 改 `src/` + 1 个 doc | 设计层 | 改完顺便更新相关文档 |
| 改 3+ docs + `src/` | **概念层** | **先写 PRD（`docs/prd/`），评审后再动手** |

概念层跳过的代价：代码改完了发现文档要改三处才能自洽，反过来暴露了设计没想清楚。

### 重构红线

以下情况**禁止**重构：
- 功能还没做完（先做功能，后理结构）
- 改动超过 2 个文件的目录/文件重命名（先列清单确认方向）
- 没有功能收益的纯重组（除非影响后续开发）

## 设计评审流程

原型或功能开发完成后，按以下流程进行评审和迭代：

### 第一阶段：理解

1. 阅读设计文档（`docs/index.md` 或对应功能文档）
2. 理解设计意图：动机、玩法、建模
3. 打开原型/实现，**不操作先看 3 秒**——记录第一印象

### 第二阶段：分析

撰写分析文档至 `docs/qa/`，覆盖以下维度：

| 维度 | 考察内容 |
|------|---------|
| 叙事层 | 设计动机是否被正确传达？用户能否感受到"这是什么游戏" |
| 玩法层 | 设计文档中的机制是否被完整实现？每个机制的可交互程度如何 |
| 交互层 | 操作路径是否流畅？是否有反馈回路断裂 |
| 视觉层 | 色彩/布局/字体是否服务于机制？是否有装饰性冗余 |
| 技术层 | 数据结构是否可扩展？实现是否有"死胡同" |

### 第三阶段：量化评估

使用五项标准打分，写入分析文档：

| 维度 | 权重 | 关键问题 |
|------|------|---------|
| 意图传达度 | 20% | 标题/副标题/首屏是否让用户秒懂 |
| 交互闭环率 | 30% | 每个操作的触发→反馈→状态更新链是否完整 |
| 信息层级清晰度 | 15% | GCW 信息块数是否在 7±2 以内 |
| 视觉-机制一致性 | 20% | 颜色/样式是否有功能含义，且含义一致贯穿 |
| 可扩展性预留 | 15% | 数据结构/代码架构是否为后续开发留了空间 |

可运行 `tests/audit.py` 量化测量视觉熵、信息密度、亮度网格。

### 第四阶段：迭代

按优先级修问题：

- **P0（阻断级）**：交互闭环断裂（如地图无响应、按钮无反馈）
- **P1（信任级）**：预期管理失败（如承诺功能不存在、选项点了没变化）
- **P2（流畅级）**：信息过载、注意力竞争、不必要的操作步骤
- **P3（增强级）**：视觉一致性、历史记录、图例辅助

每轮迭代后更新评分和文档。

### 第五阶段：归档

- 分析文档 → `docs/qa/`
- 评估工具 → `tests/`
- 评分和结论写入文档末尾

## 交互设计原则检查清单

原型修改前后，对照以下清单：

### Nielsen 10 项启发式

- [ ] 系统状态可见性：用户是否知道当前状态？
- [ ] 系统与现实匹配：术语是否符合用户心智模型？
- [ ] 用户控制与自由：是否有撤销/反选/取消？
- [ ] 一致性与标准：相同颜色是否总表示相同含义？
- [ ] 错误预防：关键操作前是否有确认步骤？
- [ ] 识别而非回忆：是否有关键信息的持久提示（图例、日志）？
- [ ] 灵活性与效率：是否有快捷操作？
- [ ] 审美与极简设计：GCW 信息块数是否超标？
- [ ] 帮助用户识别错误：空输入/非法操作是否有提示？
- [ ] 帮助文档：是否需要 onboarding？

### 游戏设计专项

- [ ] 反馈回路完整：决策 → 结果 → 状态变化 → 可验证
- [ ] 渐进式披露：信息是否按需展示而非全部堆砌
- [ ] 模式防错：不同模式下的操作是否不会互相干扰
- [ ] 叙事增强机制：每个玩法机制是否配有叙事解释
- [ ] **非对称权重**：同样的 outcome 对不同选择是否有不同代价（如防守反击失败的信心打击 > 进攻失败）
- [ ] **非玩家驱动机制**：是否有按时间触发的自动事件（如私人事务、阶段战报），让世界感觉在自行运转

## 提交规范

提交信息格式：

```
<type>: <简短描述>

type: feat / fix / docs / chore / refactor
描述用中文，50 字以内，不包含句号
```

示例：
- `docs: add interaction design report for prototype`
- `fix: map now updates after command execution`
- `chore: shorten filenames - interaction-report→interaction`

## 文档贡献规范

### AI 辅助生成文档要求

1. **避免对话残留**：删除 AI 的对话式表达（如"这就说得通了""奇怪就对了"）
2. **消除重复**：跨文件重复的分析内容只保留一份，用索引引用
3. **结论优先**：删除推导过程，只保留最终结论和可执行方案
4. **格式规范**：用表格/清单代替长段落，代码只保留关键接口
5. **分层清晰**：
   - `docs/index.md`：重构路线/核心方案
   - `docs/模块/index.md`：模块现状清单
   - 避免同一层级内容拆成多个文件

### 文档质量检查清单

- [ ] 无对话语气残留
- [ ] 跨文件重复内容已合并
- [ ] 所有分析都有对应结论/方案
- [ ] 代码块与当前实现一致（或仅保留接口）
- [ ] 用表格/清单替代长段落
