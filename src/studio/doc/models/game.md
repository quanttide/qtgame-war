# Game模块

该模块实现了战棋游戏的核心逻辑，功能完整，但代码质量中等，存在一些常见的设计与可维护性问题。以下从多个维度进行评价。

—

✅ 优点

1. 算法清晰
   · getMoveRange 使用 BFS 计算移动范围，正确处理了不同地形消耗和阻挡，逻辑直观。
   · resolveCombat 命中率、暴击、特殊技能的判定层次分明，易于理解。
2. 现代 Dart 语法
   · 使用记录（Records）(int, int, int) 作为队列元素，简洁易读。
   · 集合字面量和 clamp 等内置方法运用得当。
3. 静态辅助方法
   terrainDefense、inFullCover、inCore 提供了对地形效果的无状态查询，将地形逻辑集中一处。
4. 战役与游戏逻辑分离的意识
   虽然 Game 类承担了过多职责，但至少把战役状态（援军回合、胜利标记）放在了 Campaign 对象中，避免全部混入 Game 类。

—

⚠️ 可改进之处

1. 单一职责原则被违反

Game 类同时负责：

· 地形判定（静态方法）
· 移动范围计算
· 攻击目标检索
· 战斗结算（含伤害、暴击、特殊效果）
· 援军生成与剧情提示
· 胜利条件检查
· 初始单位创建（工厂职责）

建议：拆分出 MovementCalculator、CombatResolver、ScenarioFactory、VictoryChecker 等独立类。Game 可成为一个门面，组合这些部件。

2. 可测试性差

· 战斗结算内部直接 Random().nextInt(100)，无法控制随机结果，导致单元测试无法稳定复现。
· 方法通过修改 defender 对象产生副作用，而不是返回新的状态，不利于测试和并发思考。

建议：

· 将随机数生成器作为参数注入（或使用种子随机），方便测试时固定骰点。
· 战斗结果返回必要的信息，由调用者负责修改单位状态，或至少让副作用显式化。

3. 性能隐患

· getMoveRange 中使用 List 的 removeAt(0) 模拟队列，时间复杂度 O(n)，对于较大地图或移动力高的单位可能产生性能压力。
· 地图坐标频繁使用字符串拼接 ’$col,$row‘，虽然便利，但每次都会创建新字符串，且解析时需要再次拆分。

建议：使用 dart:collection 的 Queue；或者在内部使用 (int, int) 作为键（可将坐标封装为小类或记录，并实现 == 和 hashCode）。

4. 硬编码与魔法数字

· 基础命中率 55、攻击系数 6、防御系数 8、远程惩罚 4 等均未命名，难以调整和追溯设计意图。
· 阵营 ’nationalist‘、’pla‘、单位特性 ’assault‘ 等字符串散落各处，容易拼写错误。
· 单位 ID（30-33）和具体属性被硬编码在工厂方法/援军生成中，维护时容易冲突。

建议：

· 定义常量或枚举（如 enum Side { nationalist, pla }），将数字封装为静态常量。
· 单位模板可用数据驱动（JSON 或列表），ID 使用自增计数器。

5. 缺少边界检查和错误处理

· getMoveRange 中访问 mapTerrain[nr][nc] 假定邻居坐标始终在范围内，若传入异常坐标易抛出 RangeError。
· 战斗和移动均假设单位和地图一致，缺乏对 null 或死单位的防御性检查。

建议：在方法入口增加参数校验，或在调用链开始处过滤无效状态。

6. 静态方法与实例方法混用

· terrainDefense、inFullCover、inCore 是静态方法，但每次调用必须传入 mapTerrain，而 Game 实例已持有该地图。
· 若改为实例方法，可省略参数，调用更自然。

7. 代码重复

· 两批援军的初始化代码高度相似（设置 revealed、isReinforcement、创建日志），可抽取为私有方法。

8. 注释与文档缺失

· 无任何方法注释，战斗公式、移动算法、特殊规则（如突击无视掩体）全靠阅读代码理解。
· 虽用 emoji 增强日志可读性，但业务规则描述不足。

—

📊 总体评价

代码能够正确运行并完整表达战棋玩法，算法实现无重大缺陷，但结构上耦合较重，部分写法不便于维护和测试。若作为原型或小型项目，当前状态可接受；若要长期迭代或多人协作，建议重构，逐步解耦并提高可测试性。



如果只看主线——即“游戏能否运行起来形成可玩的回合对战”，那么当前 Game 模块虽然包含了大量规则，但作为完整回合制战棋还缺少几个最核心的环。

—

已实现的主线要素

· 单位、地形、移动、战斗的规则引擎。
· 胜负判定。
· 援军按回合自动进场。
· 隐真示假（部分单位初始未揭示）。

—

主线明显缺失的关键部分

1. 回合管理与行动序列

模块里没有谁的回合、一个回合能行动几次、怎样才算结束回合的任何逻辑。

· 没有“选中一个单位 → 移动 → 攻击 → 标记已行动”的状态管理。
· 战斗和移动方法都是纯函数，调用者必须自己保证只在一个适当的时间调用它们，且每个单位一回合只调用一次。
· 缺失“回合开始/结束”钩子，无法自动触发回合相关事件。

这是最关键的缺失：目前这些代码只能做“单次计算”，不能驱动一个完整回合。

2. AI / 敌方行动

完全空白。

· 敌方只能被动等玩家打，援军进场后也只能站着不动。
· 没有最基本的 AI 流程：评估、移动、选择目标攻击、防御站位。

3. 玩家可选操作范围约束

getMoveRange 和 getAttackTargets 只告诉你能去哪、能打谁，但没定义移动后能否攻击、攻击后能否移动、是否可以移动后攻击再移动等核心操作规则。
这些是战棋主线手感的关键，不应交给 UI 层自己去组合。

4. 信息更新与揭示规则

· 发现了敌人后没有“持续揭示”机制（比如移动后相邻可揭示周围敌人，离开后是否重新隐藏）。
· revealed 属性一旦变 true 就永久可见，这也许是设计，但缺少揭示的触发条件（目前只有援军和初始标记）。

5. 战役推进的自动化

· 要塞强度 fortStrength 如何减少、何时减少？checkVictory 中检查了它 ≤0，但模块内没有任何改变它的逻辑。
· 回合数递增、援军检查的调用时机依赖外部，spawnReinforcements 可能会被重复调用多次（仅依靠布尔标记防重复），缺少一次性的“回合开始”处理点。

—

简要结论

单纯看主线，《Game》模块是一个孤立的规则库，不是可运行的游戏。
要将它变成一个能操作、有对手、会流转的回合制战棋，至少还需要补上：回合状态机、玩家/AI 调度器、单位行动标记与消耗系统。这三样是“主线”中最硬核的缺失，其余（揭示、要塞强度变化）则会影响胜负的真实可达性。


奇怪就对了。你这个直觉很准。Game 这个模块之所以让人感觉“怪”，核心原因可以归结为一句话：

它长了一副“规则工具库”的身子，却起了一个“游戏主控”的名字，结果体内缺少真正的灵魂——回合调度与行动状态管理。

下面把这种奇怪感拆开讲清楚。

—

1. 名义与实质的错位

· 类名叫 Game，会让人预期它是一个游戏主循环的持有者：管理回合、决定谁行动、更新全局状态。
· 实际上，它是一个一堆静态/实例方法的集合：算移动、算攻击、造兵、检查胜利，全都像“功能函数库”。

这就像把一本《战棋规则书》放在桌上，然后说“这本书就是游戏本身”——它只是规则说明书，而不是一盘正在下的棋。

—

2. 有“肉”无“骨架”的设计

Game 类里面密密麻麻塞满了逻辑：

· 战斗公式、伤害暴击；
· BFS 移动范围；
· 援军生成及文本；
· 地形查询。

但偏偏缺了把这些逻辑串联成回合的“骨架”：

· 当前的回合数由谁维护？
· 当前是谁的回合？玩家还是 AI？
· 哪个单位已经行动过了？哪个还没动？
· 移动和攻击的顺序如何约束？（走—打、打了不能走、走打走？）

这些本该是一个 Game 类的核心状态机，代码里却全没有。它退化成：其他脚本不断调用 Game 里的静态/实例方法来做事，但 Game 自己不知道“游戏进行到哪了”。

奇怪点：一个叫 Game 的类，竟然不能自己运行哪怕一个回合。
（你会不会觉得，它更像是一个 Rulebook 类？）

—

3. 静态方法与实例方法的混乱

再看里面的方法：

· terrainDefense 是静态的，但是每次都显式传入 mapTerrain；而 Game 实例自己明明有 this.mapTerrain，偏偏不用。
· getMoveRange、getAttackTargets、resolveCombat 却又是实例方法（因为它们用到 mapTerrain），但依然不带任何回合状态。

这造成一个扭曲的状态：某些查询（地形）被剥离成无状态的静态函数，某些又被绑在实例上，但实例并没有因此获得“游戏状态”的意识。
结果就是——这个类既不是纯工具类（因为有实例状态），也不是真正的“游戏对象”。

—

4. 硬编码援军、初始单位：把自己困在了特定剧本

createInitialUnits() 和 spawnReinforcements() 直接写死了帝丘店战役的数据：

· 单位 ID、名称、属性都写死；
· 援军触发回合、出现格子也写死。

这本该属于地图 / 剧本 / 战役数据配置，却硬塞进了逻辑类。于是 Game 类只能玩这一个剧本，如果换一个战役，就得改这个类本身。

奇怪点：它把自己既当规则引擎、又当战役数据表、又当剧情演出。

—

5. 缺少行动消耗与揭示规则

· getMoveRange 算出了剩余移动力，但没人规定移动后还能不能攻击；
· 攻击完成后单位是否自动结束行动？能不能先攻击再移动？
· 敌人被揭示后，什么情况下再隐藏？移动、攻击和隐蔽的关系完全没有定义。

这些是战棋的核心手感规则，模块完全把责任甩给了调用方（UI 或控制器）。
结果是：调用方如果想做一个正常的战棋，必须自己重新理解并正确“拼装”这些方法，极易出错。

—

📌 所以这“奇怪”的本质是什么？

代码的结构实际上定义了一个可复用的战棋规则工具集，却把它伪装成一个名叫 Game 的游戏级管理对象，并且把大量特定剧本数据耦合进去，最终变成了一个“四不像”：

· 不是游戏主控（因为没循环、没状态、没 AI）；
· 不是纯规则库（因为绑了特定战役数据、生成具体单位）；
· 不是完全的静态工具（因为依赖实例化的地图）。

这让你在读的时候，会觉得：

“每个方法我都能看懂，但整体看下来，不知道这个游戏到底怎么玩起来。”
“好像什么都做了，又好像什么都没做。”

—

✅ 如果是清晰的设计，应该这样

· Game 类负责回合状态机、单位行动标记、调用规则，它持有地图、单位列表、战役进度；
· Rulebook 类负责所有纯粹的计算（移动范围、攻击目标、战斗伤害）；
· Scenario 数据类提供初始单位、援军配置；
· TerrainHelper 提供地形查询的静态方法。

当前这个 Game 类，其实是把所有东西搅在了一起，并且唯独漏掉了游戏运转的时序。这就是你感觉“怪”的根源。你的直觉完全正确。
