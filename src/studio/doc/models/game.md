# Game模块

该模块实现了战棋游戏的核心逻辑，功能完整，但代码质量中等，存在一些常见的设计与可维护性问题。以下从多个维度进行评价。

—

✅ 优点

1. 算法清晰
   · getMoveRange 使用 BFS 计算移动范围，正确处理了不同地形消耗和阻挡，逻辑直观。
   · resolveCombat 命中率、暴击、特殊技能的判定层次分明，易于理解。
2. 现代 Dart 语法
   · 使用记录（Records）(int, int, int) 作为队列元素，简洁易读。
   · 集合字面量和 clamp 等内置方法运用得当。
3. 静态辅助方法
   terrainDefense、inFullCover、inCore 提供了对地形效果的无状态查询，将地形逻辑集中一处。
4. 战役与游戏逻辑分离的意识
   虽然 Game 类承担了过多职责，但至少把战役状态（援军回合、胜利标记）放在了 Campaign 对象中，避免全部混入 Game 类。

—

⚠️ 可改进之处

1. 单一职责原则被违反

Game 类同时负责：

· 地形判定（静态方法）
· 移动范围计算
· 攻击目标检索
· 战斗结算（含伤害、暴击、特殊效果）
· 援军生成与剧情提示
· 胜利条件检查
· 初始单位创建（工厂职责）

建议：拆分出 MovementCalculator、CombatResolver、ScenarioFactory、VictoryChecker 等独立类。Game 可成为一个门面，组合这些部件。

2. 可测试性差

· 战斗结算内部直接 Random().nextInt(100)，无法控制随机结果，导致单元测试无法稳定复现。
· 方法通过修改 defender 对象产生副作用，而不是返回新的状态，不利于测试和并发思考。

建议：

· 将随机数生成器作为参数注入（或使用种子随机），方便测试时固定骰点。
· 战斗结果返回必要的信息，由调用者负责修改单位状态，或至少让副作用显式化。

3. 性能隐患

· getMoveRange 中使用 List 的 removeAt(0) 模拟队列，时间复杂度 O(n)，对于较大地图或移动力高的单位可能产生性能压力。
· 地图坐标频繁使用字符串拼接 ’$col,$row‘，虽然便利，但每次都会创建新字符串，且解析时需要再次拆分。

建议：使用 dart:collection 的 Queue；或者在内部使用 (int, int) 作为键（可将坐标封装为小类或记录，并实现 == 和 hashCode）。

4. 硬编码与魔法数字

· 基础命中率 55、攻击系数 6、防御系数 8、远程惩罚 4 等均未命名，难以调整和追溯设计意图。
· 阵营 ’nationalist‘、’pla‘、单位特性 ’assault‘ 等字符串散落各处，容易拼写错误。
· 单位 ID（30-33）和具体属性被硬编码在工厂方法/援军生成中，维护时容易冲突。

建议：

· 定义常量或枚举（如 enum Side { nationalist, pla }），将数字封装为静态常量。
· 单位模板可用数据驱动（JSON 或列表），ID 使用自增计数器。

5. 缺少边界检查和错误处理

· getMoveRange 中访问 mapTerrain[nr][nc] 假定邻居坐标始终在范围内，若传入异常坐标易抛出 RangeError。
· 战斗和移动均假设单位和地图一致，缺乏对 null 或死单位的防御性检查。

建议：在方法入口增加参数校验，或在调用链开始处过滤无效状态。

6. 静态方法与实例方法混用

· terrainDefense、inFullCover、inCore 是静态方法，但每次调用必须传入 mapTerrain，而 Game 实例已持有该地图。
· 若改为实例方法，可省略参数，调用更自然。

7. 代码重复

· 两批援军的初始化代码高度相似（设置 revealed、isReinforcement、创建日志），可抽取为私有方法。

8. 注释与文档缺失

· 无任何方法注释，战斗公式、移动算法、特殊规则（如突击无视掩体）全靠阅读代码理解。
· 虽用 emoji 增强日志可读性，但业务规则描述不足。

—

📊 总体评价

代码能够正确运行并完整表达战棋玩法，算法实现无重大缺陷，但结构上耦合较重，部分写法不便于维护和测试。若作为原型或小型项目，当前状态可接受；若要长期迭代或多人协作，建议重构，逐步解耦并提高可测试性。
